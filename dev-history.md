Написание инструкции по индивидуальной настройке данной ВМ для вновь прибывших разработчиков.</br>

3.	Настройка базовых конфигураций приложения.</br>
3.1.	Настройки клиента на Angular (фронтенд).</br>
3.1.1.	Указаны расположения ассетов и стилей, в том числе импортируемых из Cesium.</br>
3.1.2.	Указаны расположения файлов глобальных стилей, добавляющие возможность использование кратких путей при их импортах.</br>
3.1.3.	Добавлены и подключены environment-файлы, устанавливающие хост сборки серверной части в зависимости от вариантов сборки. Dev-сборка берет значение из глобального «location.hostname», так как обычно собирается на ВМ разработчика вместе с клиентской частью, а prod-сборка вписывается вручную, так как серверная часть может быть собрана в другом месте.</br>
3.1.4.	Вышеуказанные настройки также применены к части, описывающей сборку для тестов.</br>
3.1.5.	В объекте, описывающем dev-сборку, host клиентской части установлен, как «0.0.0.0» (в .json нельзя напрямую импортировать значение «location.hostname»), что также автоматически обеспечивает актуальное значение хоста вне зависимости от локации сборки.</br>
3.1.6.	В объекте, описывающем dev-сборку, выключен hmr-параметр (hot module replacement), так как несовместим с отложенной загрузкой компонентов в шаблонах при использовании @defer.</br>
3.1.7.	В объекте, описывающем dev-сборку, включено подробное логирование процесса сборки в консоль.</br>
3.2.	Настройки сервера на Nest.js (бэкенд).</br>
3.2.1.	В качестве http-платформы установлена «NestExpressApplication».</br>
3.2.2.	Установлен глобальный префикс url, как «api» (совпадает с началом url с фронтенда), ко всем контроллерам сервера (в них можно не указывать).</br>
3.2.3.	Задействована библиотека-сервис «ConfigService» – для взаимодействия с файлами окружения (.env) из любой точки приложения.</br>
3.2.4.	Установлены разрешения для межсетевой политики CORS (допуск любых запросов; считывание и запись в cookie – нужно для данных сессии).</br>
3.2.5.	Настроен и глобально подключен сессионный middleware (библиотека «express-session»). При этом:</br>
\- установлена директория хранения файлов сессии;</br>
\- установлено имя для идентификатора сессии в куки (для req- и res-объектов);</br>
\- используемый в настройках ключ для формирования хэша подписи уникального cookie-идентификатора сеанса сохранен в .env-файле;</br>
\- время устаревания сессии установлено на 24 часа;</br>
\- сессия обновляется при каждом обращении пользователя;</br>
\- приоритет над другими cookies данного домена (в части избегания автоочистки) для сессионных данных установлен, как высокий.</br>
3.2.6.	Установлено глобальное применение валидации запросов (из библиотеки «class-validator») на основе DTO (data transfer object). При этом настройка отправляет в консоль сервера возможные предупреждения, а также добавляет уточнение в ошибку запроса (в «error.error.message») при провале валидации.</br>
3.2.7.	Для наглядности по окончанию сборки сервера также установлено логирование в консоль его актуального хоста путем получения значения из ОС (выведено в сетевой конфиг).</br>
3.3.	Параметры межсетевого взаимодействия клиентской, серверной частей и БД.</br>
3.3.1.	На бэкенде для упрощения доступа и большего контроля над параметрами .env-файла (и прочей статикой) созданы надстройки над «ConfigService», валидирующие параметры окружения для серверной части и БД и хранящие их геттеры.</br>
3.3.2.	На фронтенде создан конфиг-файл для доступа к статике серверной части посредством использования импорта значений из ранее созданных environment-файлов.</br>
3.3.3.	Добавлен конфиг кастомных свойств (например, «mostDetailedHeights») и маршрутов для Cesium.</br>
3.4.	Настройка маршрутизации клиента приложения.</br>
3.4.1.	Корневой компонент разделен на два main-компонента: клиент «Ока» (основа приложения) и лэндинг (пока что только для тестирования маршрутизации). В последствие аналогично могут быть задействованы и другие глобальные модули проекта (имеющие собственный эндпойнт), например админпанель БД, личный кабинет пользователя, стол заказов и пр.</br>
3.4.2.	Между одноуровневыми main-компонентами, а также между компонентами лэндинга, настроена маршрутизация: по прямым ссылкам и по пользовательскому вводу URI в адресную строку браузера. Ошибки в URI учтены и приводят к логическому перенаправлению на нужный модуль. При этом для компонентов лэндинга установлена ленивая загрузка – будет осуществлена не просто по загрузке приложения, а только при переходе по адресу лэндинга.</br>
3.5.	Создание глобальных настроек стилей приложения (на базе SCSS)</br>
3.5.1.	Создание scss-конфигурации для Angular Material. При этом определены:</br>
\- набор применяемых палитр и их пресеты;</br>
\- глобальные переменные для выбранной палитры и палитры по умолчанию;</br>
\- учет выбранной пользователем в браузере темной или светлой темы;</br>
\- глобальные стили и переменные для стилей, соотносящиеся с mat-переменными (чтобы иметь возможность корректировать повсеместно примененные стили по единственному изменению);</br>
3.5.2.	Создание общих файлов для стилей согласно архитектуре приложения и связи между ними, создание заготовок под сценарии медиазапросов с использованием SCSS (удобней и лаконичней).</br>
3.6.	Настройка конфигурации компилятора TypeScript.</br>
3.6.1.	Составлен пресет параметров, обеспечивающий строгую проверку типов и кодовых конструкций с учетом обеспечения возможности применения псевдонимов в абсолютных импортах модулей (@ и @global – на фронтенде, @ – на бэкенде), а также отсутствия необходимости в указании расширения импортируемого файла.</br>
3.6.2.	Для команды разработчиков добавлены пояснительные комментарии к примененным в конфигурации параметрам.</br>

4.	На фронтенде созданы перехватчики сетевых запросов. </br>
4.1.	Автодобавление домена серверной части приложения в url запроса при наличии префикса «/api» («прокси» на бэкенд) с установкой таймаута запроса в пять секунд. Профит: разработчику не требуется каждый раз брать статику из сетевого конфига и модифицировать объект запроса при обращении на бэкенд.</br>
4.2.	Отклонение дублей http-запросов с возвратом соответствующей ошибки в консоль. Профит: автоматизация и унификация логики блокировки.</br>
4.3.	Проверка на bad html строковых данных (если «typeof req.body === ‘string’») в теле неидемпотентных запросов. Профит: дополнительная защита от XSS-атак, в подстраховку средствам Angular. Проверки тела запроса иных конструкций проводятся по месту при необходимости.</br>
4.4.	Кэширование get-запросов. Применяется по желанию путем добавления в опции запроса свойства «context: new HttpContext().set(CACHING_ENABLED_TOKEN, true)». Профит: оптимизация расходов трафика и скорости обработки сетевых запросов.</br>
4.5.	Показ возле курсора мыши mat-спинера (под «mat» подразумервается библиотека Angular Material – и далее по тексту), характеризующего ожидание ответа от сервера (с возможностью показа прогрессии при http-событиях типов UploadProgress и DownloadProgress). Использует соответствующий глобальный сервис, который дает возможность применить данную функциональность не только в перехватчике. Профит: улучшение UX.</br>

6.1.	Состав: два переиспользуемых (в модулях main-oko и лэндинга) компонента-кнопки и два соответствующих сервиса.</br>

7.	На фронтенд в основной модуль приложения main-oko (и далее по тексту так же, касаемо фронтенда, если не указано иное) добавлен Cesium.Viewer (canvas с эллипсоидом).</br>
7.1.	Состав: общий для всех компонентов main-oko сервис, плюс, монтирующая Viewer директива при первом рендеринге данного модуля, в которой также описан порядок загрузки других сервисов, относящихся к работе с Cesium.Viewer.</br>
7.2.	Настроен стартовый пресет настроек Viewer.</br>
7.2.1.	Из настроек по умолчанию исключено все лишнее. Всем опциям дано описание в виде комментариев (для команды разработчиков).</br>
7.2.2.	Добавлены кастомные установки, оптимизирующие и расширяющие функциональность Cesium.Viewer:</br>
\- хранение состояния выбранной сущности (например, для чтения кастомным infobox);</br>
\- подключение миксина, добавляющего поддержку перетаскивания для файлов CZML;</br>
\- установка начального положения камеры наблюдения над эллипсоидом (результат совпадает с возвратом вида по умолчанию с помощью соответствующей кнопки «ZnemzNavigationMixin» – см. далее по тексту);</br>
\- уменьшение количества усеченных полигонов, что позволяет увеличить производительность;</br>
\- экономия ресурсов памяти за счет игнорирования обработки рельефа вне поля зрения;</br>
\- экономия ресурсов памяти за счет отключения освещения глобуса источником света сцены (отсутствует тень на глобусе);</br>
\- включено отображение атмосферы при наблюдении у поверхности эллипсоида;</br>
\- включен максимальный зум на колесике мыши (ограничение не позволит увести камеру под рельеф);</br>
\- скрыт логотип Cesium (левый нижний угол);</br>
\- применен пользовательский хук, чтобы полилинии и примитивы рисовались всегда поверх;</br>
\- отключено дефолтное приближение к сущностям по двойному клику ЛКМ (мешает, например, на точечных объектах, плюс, добавляет (и фокусит) ненужную панель в место, где должен быть отображен отключенный стандартный infobox), вместо этого написана своя, кастомная функциональность перелета;</br>
\- отключен дефолтный выбор сущностей по ЛКМ (так как в проекте не нужен подобный вызов стандартного infobox), вместо этого написана своя, кастомная функциональность, обновляющая соответствующий параметр (состояние «pickedEntityId») объекта Viewer;</br>
\- добавлены сеттеры для базовых установок Viewer (например, ImageryProvider).</br>

8.	На фронтенд добавлена функциональность по определению координат под курсором мыши.</br>
8.1.	Состав: сервис, осуществляющий расчеты, их преобразования и строковое представление, а также компонент, включающий в себя меню выбора системы координат и два поля для отображения координат: над меню выбора системы координат и возле курсора мыши.</br>
8.2.	Получить данные сервиса (другими сервисами и компонентами) о позиции курсора мыши, выбранной системе координат и описании текущих координат возможно, обращаясь, как непосредственно к состояниям сервиса, так и к свойствам сущности с «id: 'mouse'» созданного ранее объекта Viewer.</br>
8.3.	События, вызывающие пересчет координат: движение курсора мыши по эллипсоиду (остальные позиции курсора не учитываются) и зум колесиком мыши, если ее курсор находится на эллипсоиде. При этом в целях оптимизации при отслеживании данных событий из их потока в расчеты попадает только 1/10 часть (каждые 100 мс).</br>
8.4.	Поля для отображения координат показаны только в случае нахождения курсора мыши на эллипсоиде.</br>
8.5.	Поле для отображения координат, находящееся возле курсора мыши имеет возможность вариативной видимости, которая контролируется чекбоксом поля для отображения координат, находящегося над меню выбора системы координат.</br>
8.6.	Меню выбора системы координат и поля для их отображения используют элементы и стили библиотеки Angular Material.</br>
8.7.	Полям компонента добавлены анимации плавного входа / выхода. Полю возле курсора мыши добавлена плавность его перемещения.</br>
9.	На фронтенд добавлен компонент навигационного миксина «ZnemzNavigationMixin»: компас-джойстик с кнопками («приблизить», «удалить», «возврат к виду по умолчанию») и «масштаб» (шаг тайловой сетки при текущей высоте камеры).</br>

10.	На фронтенд и бэкенд добавлена функциональность авторизации и регистрации пользователей (глобально, но пока компонент размещен только в main-oko-модуле).</br>
10.1.	Состав.</br>
10.1.1.	На фронтенде: сервис, осуществляющий запросы на бэкенд и хранящий неконфиденциальную информацию о пользователе (на текущий момент только имя пользователя), компонент авторизации / регистрации пользователей и переиспользуемый компонент, содержащий варианты использования остальной функциональности, относящейся к аккаунту пользователя (на текущий момент меню включает в себя только логаут).</br>
10.1.2.	На бэкенде: контроллер «user» и соответствующие целям модуля используемые им сервисы, взаимодействующие с БД и хранилищем сессий пользователей.</br>
10.2.	Описание фронтенда.</br>
10.2.1.	Кнопка, открывающая компонент, служащий для взаимодействия с аккаунтом пользователя, имеет вид, изменяемый (в части иконки) в зависимости от ожидаемого сценария (вход / регистрация, либо действия с аккаунтом).</br>
10.2.2.	Формы авторизации / регистрации доступны только при отсутствии авторизации пользователя. В обратном случае по нажатию той же кнопки будет отображено меню действий с аккаунтом.</br>
10.2.3.	Для форм авторизации / регистрации реализованы автофокус и управление с клавиатуры.</br>
10.2.4.	Формы авторизации / регистрации имеют блокировки своего закрытия до момента окончания осуществления сетевых запросов от используемого ими сервиса: задано соответствующее условие закрытия диалога (mat-dialog-элемент), выставлены временные блокировки самих форм и не привязанных к ним кнопок.</br>
10.2.5.	Форма регистрации сохраняет в сервис (оперативную память) введенные пользователем значения своих полей и восстанавливает их при случайном закрытии компонента. Резервные значения не записываются в local storage браузера и будут сброшены при перезагрузке страницы, а также удалены при успехе сценария регистрации пользователя.</br>

10.3.1. 10.3.5.	При логауте помимо удаления данных сессии на сервере, в ответ на клиент также закладывается удаление сессионных данных в его cookie.</br>
10.3.6.	Тело запроса на регистрацию с данными, сохраняемыми в БД, резервно проверяется на bad html.</br>
10.3.7.	Все обращения к БД используют пул запросов, что исключает их конфликт.</br>
10.3.8.	Все обращения к БД используют параметризованные запросы, что защищает БД от вредоносных sql-инъекций.</br>
